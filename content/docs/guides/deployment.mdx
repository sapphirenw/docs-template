This guide covers deploying your documentation site using Docker, and integrating it into existing projects as a git submodule.

## Monorepo / Submodule Setup

If you're adding documentation to an existing project, you can use this platform as a git submodule. This keeps the docs platform separate from your content while allowing you to pull updates.

### Add as Submodule

```bash
cd your-monorepo
git submodule add https://github.com/your-org/doc-hosting-platform.git packages/docs
```

### Project Structure

```
your-monorepo/
├── packages/
│   └── docs/                    # ← submodule (this platform)
│       ├── docs.config.json     # ← sample config (don't edit)
│       ├── content/docs/        # ← sample content (ignore)
│       └── src/
├── docs-content/                # ← YOUR documentation files
│   ├── getting-started/
│   │   └── intro.mdx
│   └── guides/
│       └── setup.mdx
└── docs.config.local.json       # ← YOUR config (gitignored by submodule)
```

### Local Config Override

The platform loads `docs.config.local.json` first if it exists, falling back to `docs.config.json`. This file is gitignored, so your config won't conflict with submodule updates.

Create `docs.config.local.json` in the submodule directory:

```json
{
  "name": "My Project Docs",
  "description": "Documentation for My Project",
  "contentPath": "../../docs-content",
  "logo": {
    "light": "/logo.svg",
    "dark": "/logo.svg"
  },
  "colors": {
    "primary": "#18E299",
    "accent": "#0d9373"
  },
  "navigation": [
    {
      "group": "Getting Started",
      "pages": ["getting-started/intro"]
    },
    {
      "group": "Guides",
      "pages": ["guides/setup"]
    }
  ],
  "topbarLinks": [],
  "footerSocials": {}
}
```

<Note>
  The `contentPath` is relative to the submodule directory. Use `../../` to reference directories in your monorepo root.
</Note>

### Pulling Updates

```bash
# Update submodule to latest
cd packages/docs
git pull origin main
cd ../..
git add packages/docs
git commit -m "Update docs platform"
```

Or update all submodules:

```bash
git submodule update --remote --merge
```

### Contributing Back

If you fix a bug or add a feature to the platform:

```bash
cd packages/docs
git checkout -b fix/my-fix
# make changes to src/ files (not content)
git add . && git commit -m "Fix something"
git push origin fix/my-fix
# Create PR on the docs platform repo
```

<Warning>
  Only commit changes to `src/`, `public/`, or platform config files. Your content and `docs.config.local.json` should stay in your monorepo, not the submodule.
</Warning>

### Docker with Submodule

When building Docker images in a monorepo, adjust the build context and copy paths:

```dockerfile
# In your monorepo Dockerfile
FROM node:20-alpine AS builder
WORKDIR /app

# Copy submodule
COPY packages/docs ./packages/docs

# Copy your content and config into the submodule location
COPY docs-content ./packages/docs/content/docs
COPY docs.config.local.json ./packages/docs/docs.config.local.json

WORKDIR /app/packages/docs
RUN corepack enable && pnpm install --frozen-lockfile
RUN pnpm build
```

Or use volume mounts in docker-compose:

```yaml
services:
  docs:
    build:
      context: ./packages/docs
    volumes:
      - ../docs-content:/app/content/docs:ro
      - ../docs.config.local.json:/app/docs.config.local.json:ro
```

## Standalone Docker Deployment

For standalone deployments (not as a submodule), use the following setup.

### Quick Start

Build and run with Docker Compose:

```bash
docker compose up --build -d
```

Your docs are now live at `http://localhost:3000`.

## Production Build

### Build for Production

```bash
pnpm build
```

This generates an optimized standalone build in `.next/standalone` (~50MB vs ~500MB for full node_modules).

<Note>
  The `output: 'standalone'` option in `next.config.mjs` is already configured for optimal Docker builds.
</Note>

## Docker Configuration

### Dockerfile

The included multi-stage Dockerfile creates minimal production images:

```dockerfile
# Stage 1: Dependencies
FROM node:20-alpine AS deps
WORKDIR /app
RUN corepack enable && corepack prepare pnpm@latest --activate
COPY package.json pnpm-lock.yaml ./
RUN pnpm install --frozen-lockfile

# Stage 2: Builder
FROM node:20-alpine AS builder
WORKDIR /app
RUN corepack enable && corepack prepare pnpm@latest --activate
COPY --from=deps /app/node_modules ./node_modules
COPY . .
ENV NEXT_TELEMETRY_DISABLED=1
RUN pnpm build

# Stage 3: Runner (minimal)
FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1

RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs

COPY --from=builder /app/public ./public
COPY --from=builder /app/.next/standalone ./
COPY --from=builder /app/.next/static ./.next/static
COPY --from=builder /app/content ./content
COPY --from=builder /app/docs.config.json ./

USER nextjs
EXPOSE 3000
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"

CMD ["node", "server.js"]
```

### Docker Compose

```yaml
services:
  docs:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    restart: unless-stopped
    environment:
      - NODE_ENV=production
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:3000"]
      interval: 30s
      timeout: 10s
      retries: 3
```

## Image Size Optimization

### Current Optimizations

| Optimization | Savings |
|-------------|---------|
| Alpine base image | ~900MB → ~180MB |
| Multi-stage build | Excludes dev dependencies |
| Standalone output | ~500MB → ~50MB |
| .dockerignore | Excludes unnecessary files |

### Final Image Size

The production image is approximately **180-200MB**, compared to 1GB+ for naive Docker builds.

<Tip>
  Run `docker images` to verify your image size after building.
</Tip>

<Steps>
  <Step title="Build the image">

  </Step>
  <Step title="Run the container">
    ```bash
    docker run -d -p 3000:3000 --name docs docs:latest
    ```
  </Step>
  <Step title="Verify it's running">
    ```bash
    curl http://localhost:3000
    ```
  </Step>
</Steps>

## Environment Variables

Pass environment variables at runtime:

```bash
docker run -d \
  -p 3000:3000 \
  -e NEXT_PUBLIC_SITE_URL=https://docs.example.com \
  docs:latest
```

Or use an env file:

```bash
docker run -d -p 3000:3000 --env-file .env.production docs:latest
```

<Warning>
  Build-time variables (prefixed with `NEXT_PUBLIC_`) must be set during `docker build`, not `docker run`.
</Warning>

For build-time variables:

```bash
docker build --build-arg NEXT_PUBLIC_SITE_URL=https://docs.example.com -t docs:latest .
```

## Reverse Proxy Setup

### Nginx

```nginx
server {
    listen 80;
    server_name docs.example.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
```

### Traefik

Add labels to your Docker Compose:

```yaml
services:
  docs:
    # ... other config
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.docs.rule=Host(`docs.example.com`)"
      - "traefik.http.routers.docs.tls.certresolver=letsencrypt"
```

## CI/CD with GitHub Actions

```yaml
name: Build and Push

on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ghcr.io/${{ github.repository }}:latest
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

## Updating Content

To update documentation without rebuilding the entire image, mount the content directory:

```yaml
services:
  docs:
    # ... other config
    volumes:
      - ./content:/app/content:ro
      - ./docs.config.json:/app/docs.config.json:ro
```

If using a local config override:

```yaml
services:
  docs:
    volumes:
      - ./my-docs-content:/app/content/docs:ro
      - ./docs.config.local.json:/app/docs.config.local.json:ro
```

<Note>
  Content changes require a container restart when using static generation. For live updates, consider using ISR (Incremental Static Regeneration).
</Note>

## Health Checks

The included health check verifies the server is responding:

```bash
# Check container health
docker inspect --format='{{.State.Health.Status}}' docs

# View health check logs
docker inspect --format='{{json .State.Health}}' docs | jq
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Container exits immediately">
    Check logs for errors:
    ```bash
    docker logs docs
    ```
    Common causes: missing environment variables, port conflicts, or build errors.
  </Accordion>
  <Accordion title="Build fails with memory error">
    Increase Docker memory limit or add swap:
    ```bash
    docker build --memory=4g -t docs:latest .
    ```
  </Accordion>
  <Accordion title="Content not updating">
    If using volume mounts, restart the container:
    ```bash
    docker compose restart docs
    ```
  </Accordion>
</AccordionGroup>
